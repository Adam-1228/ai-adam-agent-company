# Memory - 박실행

## 추적하는 상태

- 등록된 정기 작업 명세 목록 (작업코드, 케이스ID, 스케줄, 채널, 템플릿ID, 멱등성 키 패턴)
- 오늘 실행 카운터: 성공/실패/스킵/거절
- 최근 24시간 실패 로그 (재시도 큐 결정용)
- 멱등성 키 캐시 (TTL 30일)

## 저장 위치

- 작업 명세: `tasks/active.md` 내 `OPS-*` 항목 + EC2 시스템디 타이머 정의
- 실행 로그: `reports/ops_daily_YYYY-MM-DD.jsonl` (gitignore)
- 멱등성 키 캐시: 외부 스토어 (Redis/SQLite, EC2 로컬)
- 일일 요약: `reports/ops_summary_YYYY-MM-DD.md` (gitignore)

## 갱신 주기

- 매 작업 실행 시점 (실시간)
- 매일 23:00 일일 요약 작성
- 매주 일요일 자정 멱등성 키 캐시 만료분 삭제

## 다른 에이전트와 공유하는 메모리

- 정총괄: 일일 요약 (성공/실패/스킵 카운트, 실패 상세 5건 이내)
- 최분석: 주간 익명 집계 JSONL (개인정보 없음, 케이스ID/실행시각/결과만)
- 이용대: 전송 실패한 고객 ID 목록 (회수 응대용)
- 김은보: 멱등성 충돌 또는 미등록 작업 코드 발생 시 알림

## 절대 기억하면 안 되는 것

- 응대 본문 (템플릿 ID와 변수만, 렌더링된 텍스트는 로그에 풀텍스트로 남기지 않음 — 해시만)
- 고객 개인정보 (이름/번호는 케이스ID·예약ID로 치환)
- 결제 정보 일체
- 자유 텍스트 입력 이력 (받지 않으므로 메모리에 들어올 일이 없음)

## 누적 주의사항

- 스케줄러 시각은 항상 `Asia/Seoul` 고정. UTC로 등록된 작업은 거절.
- 카카오 알림톡은 발송 실패 시 SMS 폴백 1회만 허용. 폴백 후 다시 실패하면 즉시 이용대에게 회수 응대 요청.
- 멱등성 키는 `{task_code}-{case_id}-{target_id}-{schedule_key}` 형식 강제. 충돌 시 `SKIPPED`.
- DRY_RUN=true 환경에서는 모든 외부 채널 호출을 mock으로 처리하고 로그에 `dry_run: true` 명시.
- 작업이 5분 이상 지연 시작되면 `late_start: true` 플래그 + 정총괄 알림.
