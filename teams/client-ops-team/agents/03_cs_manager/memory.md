# Memory - 이용대

## 추적하는 상태

- 진행 중 에스컬레이션 케이스 (P0/P1) 목록
- 케이스별 응대 회차 (3회 이상은 자동 P0)
- 고객별 톤 카드 (김은보가 제공)
- 즉시 에스컬레이션 키워드 사전 (월 1회 갱신)
- 자동응답 사후 검수에서 정총괄이 지적한 패턴 (학습)

## 저장 위치

- 케이스 상태: `tasks/active.md` 내 `CS-*` 항목
- 응대 이력: `reports/cs_log_YYYY-MM-DD.jsonl` (gitignore)
- 톤 카드: 고객별 `agents/03_cs_manager/tone_cards/{case_id}.md` (gitignore — 고객명 포함)
- 키워드 사전: `agents/03_cs_manager/escalation_keywords.md` (이 폴더 내, 공개 가능)
- 정총괄 피드백 학습: 본 memory 하단 누적 주의사항에 1줄씩 추가

## 갱신 주기

- 신규 인입 시점 (실시간)
- 에스컬레이션 발생 시점 (실시간, Telegram 동시 발송)
- 매일 23:30 — 그날의 자동응답 검수 결과 반영
- 매주 금요일 16:00 — 키워드 사전 검토 (정총괄 합의 후 갱신)

## 다른 에이전트와 공유하는 메모리

- 정총괄: 모든 자동응답 초안의 검수 큐 + 즉시 에스컬레이션 카드
- 김은보: 톤 카드에 없는 신규 상황 → 톤 카드 보강 요청
- 박실행: 회수 응대 완료 시 결과 회신 (실패 로그 클로즈용)
- 최분석: 주간 응대 카테고리 집계 (개인정보 없음, 카테고리/건수만)

## 절대 기억하면 안 되는 것

- 고객 응대 본문 풀텍스트 (해시 + 카테고리만, 본문은 케이스ID로 외부 보관)
- 주민번호, 카드번호, 진단명, 처방내역
- 고객의 가족/지인 개인정보 (응대 중 언급되더라도 즉시 마스킹)
- 다른 고객사 정보 (케이스ID 외 식별자)

## 누적 주의사항

- "환불"이라는 단어는 문맥과 무관하게 무조건 P0. "환불 안 받아도 돼요"도 P0 (정총괄 1회 확인).
- 새벽 인입에 자동응답 절대 금지. 분류만 하고 다음 영업일까지 보류.
- "병원 안 가도 되나요" 같은 의료 자문성 질문은 절대 답하지 않음. "담당 의료진과 상의 필요"만.
- 알림톡 템플릿은 카카오 사전심사 통과한 것만 사용. 자유 텍스트 본문은 SMS/이메일 폴백에서만.
- 동일 고객이 같은 날 3회 인입 시 자동 P0 승격. 무조건 사람.
