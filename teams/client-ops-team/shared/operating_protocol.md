# Operating Protocol — Client Ops Team

## 공통 업무 규칙

1. 각 에이전트는 자기 역할 밖의 결정을 확정하지 않는다.
2. 불확실한 정보는 `확인 필요`로 표시한다.
3. 다음 에이전트가 이어서 일할 수 있게 근거와 데이터를 남긴다.
4. 외부 발송(고객, Adam, 커머스 팀, 외부 시스템)은 정총괄 게이트를 거치기 전까지 하지 않는다.
5. 모호하면 보수적으로, 모르면 `확인 필요`로, 위험하면 정총괄에게.

## 산출물 공통 형식

모든 에이전트는 산출물 마지막에 아래 블록을 남긴다.

```text
HANDOFF
- 다음 담당:
- 넘길 자료:
- 확인 필요:
- 하드 스톱:
- 추천 액션:
```

박실행은 예외로, 정형 JSON에 동일 의미의 필드를 포함한다 (`next_owner`, `review_owner`, `failures` 등).

## 상태 값

- `TODO`: 아직 시작 전
- `IN_PROGRESS`: 진행 중
- `BLOCKED`: 외부 확인 필요
- `READY_FOR_REVIEW`: 다음 에이전트 또는 정총괄이 검토 가능
- `DONE`: 완료
- `REJECTED`: 거절 (사유 명시)

## 결정 값

- `PROCEED`: 다음 단계 진행 가능
- `REVIEW`: 정총괄 검수 후 진행
- `HOLD`: 지금은 보류 (재작업 또는 외부 확인)
- `ESCALATE`: 사람(Adam)이 결정
- `KILL`: 폐기 권장

## 파일 사용 규칙

- 업무 배정은 각 에이전트의 `inbox.md`에 기록한다.
- 누적 주의사항은 `memory.md`에 기록한다.
- 반복 가능한 양식은 `output_template.md`를 따른다.
- 실제 운영 로그(`reports/*`)는 gitignore 대상. 저장소에는 양식과 페르소나만.

## 안전 디폴트

- `DRY_RUN=true` 환경에서 외부 채널 호출은 mock. 실제 발송 전 정총괄 1회 검수 + DRY_RUN 해제는 Adam 결정.
- 의료/법률/세무/개인정보 키워드 감지 시 자동응답 즉시 중단.
- 멱등성 키 없는 자동 실행은 거절.
- 양방향 대화는 이용대만 (단, 톤 카드 범위 내).

## 책임 라인

- 일상 운영의 책임: 정총괄
- 외부 산출물의 책임: 정총괄
- 6대 케이스의 책임: Adam (정총괄이 통지)
- 자기 페르소나 준수의 책임: 각 에이전트

## 프로토콜 변경

본 문서 변경은 정총괄 1차 검토 + Adam 최종 승인. 커밋 메시지에 변경 사유 명시.
